Резюме проєкту

Що це?

Status Check — корпоративний веб-застосунок для спільного відстеження зобов'язань і дедлайнів у команді. Аналог Trello/Jira, але спрощений і орієнтований на швидку фіксацію домовленостей: хто, що, до коли, у якому проєкті.

Яку проблему вирішує?

У команді домовленості розкидані по чатах, листах і головах. Status Check — єдине місце, де видно всі відкриті зобов'язання, хто відповідальний, який статус, чи не прострочено. Це особливо важливо для менеджерів і ревьюерів, яким потрібен швидкий огляд без зайвих нарад.

1 Структура проєкту

1.1 Загальна архітектура

Проєкт — це fullstack веб-застосунок, побудований за принципом монорепозиторію: фронтенд і бекенд знаходяться в одній кодовій базі Next.js. Усі API-маршрути — це serverless-функції, які виконуються на Vercel. База даних — PostgreSQL на хмарному сервісі Neon.

2 Технологічний стек і причини вибору

2.1 TypeScript замість JavaScript або Python

ПитанняРішенняПричина
TypeScript vs JavaScript TypeScript Статична типізація — менше runtime-помилок. IDE підказує типи Prisma-моделей, API-відповідей, Zod-схем. Рефакторинг безпечніший. Для командного проєкту — обов'язковий стандарт.
TypeScript vs Python TypeScript Проєкт — веб-застосунок з фронтендом. Python потребував би окремого бекенду (Flask/FastAPI) + окремого фронтенду (React). TypeScript+Next.js — єдина мова для обох шарів, один деплой, менша складність.

2.2 Next.js 14 (App Router)

АльтернативаЧому Next.js краще для цього проєкту
React + Express Next.js = фронтенд + бекенд в одному проєкті. Немає потреби в окремому Express-сервері.
Vite + окремий API Два репозиторії, два деплої. Next.js = один деплой на Vercel.
Nuxt/SvelteKit Next.js — найпопулярніший React-фреймворк, кращий ринок праці, глибока інтеграція з Vercel.

App Router (Next.js 14) використовується замість Pages Router, оскільки підтримує Server Components, краще кешування та вбудовані layout-ієрархії.

2.3 Prisma ORM + Neon PostgreSQL

КомпонентПричина вибору
Prisma (замість Drizzle, Sequelize, raw SQL) Type-safe запити — TypeScript автодоповнення для моделей. Автоматичні міграції. Зрозумілий schema-файл. Офіційна підтримка Neon.
Neon PostgreSQL (замість Supabase, PlanetScale) Serverless PostgreSQL — немає витрат у режимі простою. Безкоштовний tier для MVP. Офіційний адаптер для Prisma. Швидкий cold start завдяки connection pooling.
PostgreSQL (замість MongoDB) Реляційна структура даних (Users, Projects, Items зі зв'язками). Сильна консистентність. SQL — стандарт ринку.

2.4 JWT + bcryptjs (аутентифікація)

КомпонентПричина
JWT (jose) Stateless-аутентифікація — не потрібна сесійна БД. Токени зберігаються в httpOnly cookies (захист від XSS). Строк дії 30 днів.
bcryptjs Стандарт хешування паролів. 10 раундів соління — захист від brute-force та rainbow tables.
Замість NextAuth/Auth.js Простіший контроль. Немає зайвих залежностей. Для MVP з одним методом входу (email+password) власна реалізація достатня.

2.5 Tailwind CSS

АльтернативаЧому Tailwind
CSS Modules Tailwind — швидша розробка, утиліт-класи прямо в JSX, не потрібні окремі .css файли.
Material UI / Chakra UI Готові бібліотеки — жорсткий дизайн. Tailwind — повний контроль над виглядом, унікальний UI.
Styled Components Tailwind — краща продуктивність (немає CSS-in-JS runtime), менший bundle.

2.6 chrono-node (NL-парсинг дат)

АльтернативаЧому chrono-node
OpenAI / Claude API chrono-node — офлайн, безкоштовно, без API-ключа. Для парсингу дат з тексту — достатньо бібліотеки, не потрібен LLM.
date-fns parse date-fns потребує точного формату. chrono-node розуміє природну мову: "через тиждень", "наступна п'ятниця", "30 червня".

Для підтримки української мови реалізована таблиця перекладів: "через тиждень" → "in 1 week", "наступного понеділка" → "next monday" тощо. Після перекладу рядок передається в chrono-node.

2.7 Zod (валідація)

Zod — runtime-валідація + TypeScript-типи з одного джерела. Схеми визначені в src/lib/validations.ts і використовуються як на вході API, так і у формах. Помилки валідації повертаються клієнту у структурованому вигляді.

2.8 Vercel (деплой)

Vercel = нативна платформа для Next.js. Автодеплой з git push, безкоштовний tier, глобальний CDN, вбудовані Serverless Functions для API-маршрутів. Альтернативи (Railway, Render) складніші в налаштуванні для Next.js.

3 Принцип роботи

3.1 Загальний потік

Браузер  →  Next.js Middleware  →  API Route Handler  →  Prisma  →  Neon PostgreSQL
                    ↑
              Перевірка JWT
              з httpOnly cookie

3.2 Аутентифікація (покроково)

  1. Реєстрація (POST /api/auth/register): email + password → bcrypt hash → збереження в БД → підписання JWT → встановлення httpOnly cookie.
  2. Вхід (POST /api/auth/login): email + password → пошук у БД → bcrypt.compare → новий JWT → cookie.
  3. Кожен запит: src/middleware.ts зчитує cookie → верифікує підпис JWT → якщо невалідний — 401/редирект на /login.
  4. Вихід (POST /api/auth/logout): видалення cookie.
  5. Клієнт: AuthContext.tsx зберігає стан поточного користувача, отриманий з GET /api/auth/me при завантаженні застосунку.

3.3 Ролі та дозволи

РольМожливості
Guest (не залогінений) Перегляд тільки (якщо дозволено), немає редагування
User (залогінений) Повний CRUD для своїх завдань (де є creator, assignee або reviewer = поточний користувач)
Admin (isAdmin=true) Повний CRUD для всіх завдань, незалежно від автора

Перевірка прав — на рівні API-хендлера: токен декодується, знаходиться User у БД, порівнюється isAdmin та поля assignee/creator/reviewer.

3.4 CRUD завдань

  1. GET /api/items: фільтрація по status та month, запит до Prisma, повернення JSON.
  2. POST /api/items: Zod-валідація тіла → запис у БД → повернення створеного item.
  3. PATCH /api/items/[id]/status: швидка зміна тільки статусу (для drag або кнопок у картці).
  4. PUT /api/items/[id]: оновлення всіх полів з перевіркою прав.
  5. DELETE /api/items/[id]: видалення з перевіркою прав.

3.5 Smart Input (NL-парсинг)

Користувач вводить текст:
"Перевірити звіт Петра до п'ятниці, проєкт Фінанси"
         ↓
POST /api/ai/parse  → src/lib/parse-task.ts
         ↓
1. Regex-пошук ключових слів (проєкт, відповідальний, рецензент)
2. Переклад українських дат: "до п'ятниці" → "till friday"
3. chrono-node.parseDate() → конкретна дата
4. Повернення JSON: { title, deadline, project, assignee, reviewer }
         ↓
Поля автоматично заповнюються у формі

3.6 Інтернаціоналізація (i18n)

Кастомна реалізація без зовнішніх бібліотек. LanguageContext.tsx зберігає поточну локаль (uk/en). Функція t(key) з src/lib/i18n.ts повертає переклад з 190+ ключів. Перемикач мови у шапці — миттєва зміна без перезавантаження.

4 Що можна покращити / недоліки

4.1 Технічні недоліки

1. Поля assignee/reviewer — рядки, не FK

Проблема: поля assignee та reviewer в StatusItem зберігаються як вільний текст, а не як зовнішній ключ до таблиці User. Якщо користувач змінить ім'я або буде видалений — дані не синхронізуються.

Покращення: замінити на assigneeId/reviewerId з @relation до User. Додати каскадне оновлення.

2. Відсутнє кешування на бекенді

Проблема: кожен запит GET /api/items — це новий SQL-запит до Neon. При великій кількості користувачів — зростання навантаження.

Покращення: додати Redis (Upstash) або Next.js fetch cache з revalidation. Або оптимістичне оновлення UI без зайвих refetch.

3. NL-парсинг — тільки дати, без семантики

Проблема: chrono-node розуміє дати, але не може виділити виконавця з тексту типу "перевірити звіт Івана" (не знає, що "Іванa" — assignee).

Покращення: інтегрувати легкий LLM-виклик (Claude Haiku) для складного NL-парсингу.

4. Відсутнє rate limiting

Проблема: API-маршрути не захищені від зловживань (brute-force login, масові запити).

Покращення: додати Vercel Edge Rate Limiting або middleware з лічильником спроб.

5. Відсутні автоматичні тести

Проблема: є тільки Playwright (E2E), але немає unit-тестів для бізнес-логіки (auth.ts, parse-task.ts, validations.ts).

Покращення: додати Vitest для unit-тестів критичних модулів.

4.2 Функціональні обмеження (для MVP — прийнятно)

  • Немає сповіщень про наближення дедлайну (email/push).
  • Немає коментарів/обговорення до завдань.
  • Немає прикріплення файлів.
  • Немає пагінації (при великій кількості завдань — повільне завантаження).
  • Статус "прострочено" виставляється вручну, а не автоматично за cron.
  • Відсутній аудит-лог змін (хто і коли змінив статус).

4.3 Архітектурні рішення на майбутнє

  • WebSocket / SSE — для real-time оновлень між колегами без ручного F5.
  • Мікросервіси — при масштабуванні NL-парсинг можна виокремити в окремий сервіс.
  • RBAC через БД — поточна роль (isAdmin bool) не масштабується на складні права; краще — таблиця roles/permissions.

5 Що було розроблено

5.1 Загальний обсяг

З нуля побудований повноцінний fullstack MVP: від схеми бази даних до деплою у продакшн на Vercel. Весь код написаний вручну — без використання готових шаблонів чи boilerplate-генераторів.

5.2 Бекенд

  • REST API — 12 ендпоінтів (auth, items CRUD, AI parse, projects, users).
  • Аутентифікація — реєстрація/вхід/вихід, JWT у httpOnly cookies, верифікація на кожному запиті.
  • Авторизація — RBAC: Guest / User / Admin з перевіркою у хендлерах.
  • Middleware — захист маршрутів на рівні Next.js middleware (до рендерингу сторінки).
  • Валідація — Zod-схеми для всіх вхідних даних, структуровані повідомлення про помилки.
  • ORM — Prisma з 5 міграціями, seed-скрипт для демо-даних.

5.3 Фронтенд

  • Дашборд — адаптивний 3/2/1-колонковий layout.
  • Список завдань — картки з кольоровими статусами, вбудований detail-overlay.
  • Фільтрація — по статусу, проєкту, виконавцю, рецензенту (desktop sidebar + mobile drawer).
  • Календар — місячний вид з кольоровими крапками по статусах для кожного дня.
  • Статистика — donut chart (розподіл за статусами), лічильники завдань.
  • Smart Input — введення завдання природною мовою з автозаповненням полів.
  • Ручна форма — повна форма з всіма полями, datepicker, combobox для вибору виконавців.
  • Темна/світла тема — повна підтримка, збереження вибору.
  • Інтернаціоналізація — повний переклад UI на українську та англійську (190+ рядків).
  • Toast-сповіщення — успіх/помилки при CRUD-операціях.
  • Scroll to top — кнопка при прокрутці довгого списку.

5.4 DevOps / Інфраструктура

  • Деплой на Vercel з автодеплоєм при push в main.
  • База даних Neon PostgreSQL у хмарі (serverless, connection pooling).
  • Змінні середовища через Vercel Dashboard (не в репозиторії).
  • Favicon — динамічний SVG-градієнт, згенерований через app/icon.tsx.
  • Автоматичний скріншотинг через Playwright (scripts/screenshot.mjs).
