# MVP → Product Roadmap

Аналіз поточного стану та що потрібно зробити для перетворення MVP в реальний продукт.

## Поточний стан MVP

- Next.js App Router + Prisma + PostgreSQL
- JWT-авторизація (cookie, 30д), прапорець `isAdmin`
- CRUD задач (items) зі статусами: TO_CHECK, EXPIRED, DONE, NOT_ACTUAL, IDEAS_BACKLOG
- Проєкти (тільки назва, без членів)
- AI-парсинг задач із природної мови
- Календарний вигляд + дашборд з фільтрами
- i18n (uk/en), dark mode
- Права на редагування: admin або creator/assignee/reviewer за іменем рядка

---

## 1. Права доступу та ролі

**Проблема:** Зараз бінарно — admin/не-admin. Немає гранульованих ролей.

- [ ] Замінити `isAdmin: Boolean` на `role: Enum` — `ADMIN | MANAGER | MEMBER | VIEWER`
- [ ] Права по ролях:
  - `ADMIN` — повний доступ, управління користувачами
  - `MANAGER` — створення проєктів, призначення задач, зміна будь-якого статусу
  - `MEMBER` — створення задач, редагування своїх + призначених
  - `VIEWER` — тільки читання
- [ ] Middleware перевіряє роль, а не просто факт авторизації
- [ ] API-ендпоінти захищені відповідно до ролі, а не тільки `getSession()`

---

## 2. Відділи (Departments)

**Проблема:** Немає концепції відділів. Всі користувачі в одному просторі.

- [ ] Модель `Department` — id, name, description
- [ ] Зв'язок `User ↔ Department` (many-to-many або many-to-one)
- [ ] UI для адміна: список відділів, прикріплення користувачів
- [ ] Фільтрація задач за відділом
- [ ] Проєкти можуть належати відділу

---

## 3. Проєкти — повноцінні, з членами

**Проблема:** `Project` — просто назва-рядок. Немає членства, немає контексту.

- [ ] Модель `ProjectMember` — userId, projectId, role (OWNER | MEMBER | VIEWER)
- [ ] Задачі видно тільки членам проєкту (або всім — налаштовується)
- [ ] UI управління проєктом: додати/видалити членів, змінити роль
- [ ] Сторінка проєкту з власним дашбордом та календарем

---

## 4. Зв'язок assignee/reviewer з реальними User

**Проблема:** `assignee` і `reviewer` — вільні рядки. Немає FK до `User`. Порівняння по `item.assignee === user.name` — крихке.

- [ ] Додати поля `assigneeId: String?` та `reviewerId: String?` як FK до `User`
- [ ] `creatorId: String` замість `creator_name: String`
- [ ] Міграція існуючих даних (matche по name → id)
- [ ] Запити через join, а не string-порівняння

---

## 5. Адмін-панель

**Проблема:** Немає UI для управління користувачами та організацією.

- [ ] Сторінка `/admin/users` — список користувачів, зміна ролі, деактивація
- [ ] Сторінка `/admin/departments` — CRUD відділів, прикріплення юзерів
- [ ] Сторінка `/admin/projects` — всі проєкти, статистика
- [ ] Захист `/admin/*` — тільки ADMIN

---

## 6. Сповіщення

**Проблема:** Нічого не відправляється при призначенні, зміні статусу, підході дедлайну.

- [ ] Email-сповіщення (nodemailer або Resend):
  - призначено задачу
  - дедлайн завтра
  - статус змінено
- [ ] In-app notification center (bell icon) з непрочитаними
- [ ] Налаштування сповіщень per-user (що отримувати, що ні)

---

## 7. Коментарі та обговорення

**Проблема:** Немає способу обговорити задачу всередині системи.

- [ ] Модель `Comment` — id, itemId, userId, body, created_at
- [ ] API CRUD для коментарів
- [ ] UI — тред коментарів в деталях задачі
- [ ] @mention користувачів у коментарях → сповіщення

---

## 8. Аудит-лог / Історія змін

**Проблема:** Немає трекінгу "хто що змінив і коли".

- [ ] Модель `AuditLog` — itemId, userId, action, oldValue, newValue, created_at
- [ ] Записувати: зміна статусу, редагування полів, призначення
- [ ] UI — вкладка "Історія" в деталях задачі

---

## 9. Пошук

**Проблема:** Немає пошуку. Фільтр тільки по статусу та місяцю.

- [ ] Full-text search по title та description (PostgreSQL `tsvector` або pg_trgm)
- [ ] Пошуковий рядок у хедері
- [ ] Розширені фільтри: по assignee, reviewer, creator, проєкту, діапазону дат, пріоритету

---

## 10. Пагінація та продуктивність

**Проблема:** `findMany` без ліміту — поверне всі тисячі записів одразу.

- [ ] Cursor-based пагінація на `/api/items`
- [ ] Infinite scroll або кнопка "Завантажити ще" на дашборді
- [ ] Індекси на `assigneeId`, `reviewerId`, `creatorId` після переходу на FK

---

## 11. Управління сесіями та безпека

**Проблема:** JWT на 30д без refresh, немає rate limiting, немає revocation.

- [ ] Refresh token + access token (access 15хв, refresh 30д)
- [ ] Таблиця `Session` для revocation (logout з усіх пристроїв)
- [ ] Rate limiting на `/api/auth/*` (через middleware або upstash/ratelimit)
- [ ] Forgot password flow з email
- [ ] Email verification при реєстрації

---

## 12. Пріоритети та теги

**Проблема:** Немає пріоритизації та класифікації задач.

- [ ] Поле `priority: Enum` — HIGH | MEDIUM | LOW | CRITICAL
- [ ] Модель `Tag`, зв'язок many-to-many з `StatusItem`
- [ ] Фільтр по пріоритету та тегах
- [ ] Сортування по пріоритету на дашборді

---

## 13. M'яке видалення (Soft Delete)

**Проблема:** Delete — назавжди. Немає кошику.

- [ ] Поле `deleted_at: DateTime?` на `StatusItem`
- [ ] Фільтр `WHERE deleted_at IS NULL` скрізь
- [ ] Сторінка "Архів" для відновлення
- [ ] Авто-очищення через N днів (cron job)

---

## 14. Аналітика та звіти

**Проблема:** Немає звітності, тільки базовий StatsPanel.

- [ ] Графіки: задачі по статусах у часі, виконання по виконавцях
- [ ] Звіт по дедлайнах: скільки прострочено, у кого
- [ ] Експорт у CSV/Excel
- [ ] Дашборд для менеджера: навантаження по команді

---

## 15. Інтеграції

- [ ] Slack/Telegram бот: отримання сповіщень, зміна статусу командою
- [ ] Webhook outgoing: при зміні статусу POST на зовнішній URL
- [ ] Календарний export (iCal) для дедлайнів

---

## 16. Тести та CI/CD

**Проблема:** Немає жодних тестів.

- [ ] Unit-тести для `parse-task.ts`, `validations.ts`, `auth.ts`
- [ ] Integration-тести для API-ендпоінтів (з тестовою БД)
- [ ] E2E-тести критичних флоу (Playwright)
- [ ] GitHub Actions: lint → typecheck → test → build

---

## Пріоритети впровадження

| Пріоритет | Завдання | Чому |

| 🔴 Критично | Ролі (RBAC) | Без цього не можна розгортати на команду |

| 🟠 Важливо | Коментарі | Без них команда веде обговорення поза системою |
| 🟠 Важливо | Email-сповіщення | Інакше задачі просто забуваються |
| 🟡 Потрібно | Аудит-лог | Потрібен для accountability |
| 🟡 Потрібно | Пошук | При 200+ задачах без нього незручно |
| 🟡 Потрібно | Soft delete | Захист від випадкових видалень |
| 🟢 Бажано | Аналітика та звіти | Для менеджерів |
| 🟢 Бажано | Інтеграції (Slack/Telegram) | Для зручності |
